# ML Delivery

A model is an engineering component with a probabilistic core. Deliver it like
engineering; evaluate it like science.

## Steps

1. **Build the baseline first.** The simplest credible non-ML solution — a
   heuristic, a lookup, a linear rule — implemented and measured. The model
   must beat it by a margin that justifies its lifetime cost; "beats nothing"
   is not a result. Sometimes the baseline ships and the workstream ends
   there; that is success, not failure.
2. **Agree evaluation before training.** The metric, the validation split
   (leakage-checked, time-aware where the world drifts), the slice-level
   breakdowns that matter to the client, and the threshold at which the model
   earns production. Criteria designed after training are alibis.
3. **Make training reproducible.** Versioned data snapshots, pinned
   dependencies, fixed seeds, and configs in code — any reported number can be
   regenerated by another engineer from the repo alone.
4. **Ship the model as a service with a fallback.** A served interface with a
   stated latency budget, input validation against the training distribution's
   assumptions, and a defined behavior when the model is unavailable or
   uncertain — typically the baseline from step 1, which is why it exists.
5. **Monitor for drift, not just uptime.** Input distribution shifts, output
   distribution shifts, and (where labels arrive) live metric tracking against
   the evaluation. Alert thresholds and the retraining trigger are set at
   launch.
6. **Write the model card.** Training data and its window, evaluation results
   with slices, known failure modes, and explicit non-uses — what this model
   must not be used for.

## Quality bar

- The baseline exists, is measured, and serves as the fallback.
- Every reported metric is regenerable from versioned code and data.
- Serving meets its latency budget with the fallback path tested.
- Drift monitoring is live before the model takes real traffic.

## Output format

Baseline comparison, evaluation report with slices, reproducible training
code, a served endpoint with fallback and monitoring, and the model card.
